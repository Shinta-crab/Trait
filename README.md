## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情
　このアプリのアイデアは現在進行形で行なっている我が家の家づくり計画の中でのちょっとした煩わしさや不満を感じたことがきっかけ生まれました。
具体的なエピソードとしては以下です。
近年の家づくりにおいて、従来の家づくりと違う点はInstagramやPinterestのようなSNSの普及によりおしゃれな家の手本となる画像が豊富にあることでそれらの画像を元にハウスメーカーや工務店に『こんな感じにできますか？』とより具体的なイメージを提供できる様になりったことが挙げられます。
　我が家もその例外ではなく、SNSから拾ってきた画像などを元にハウスメーカーに相談したりしています。
　一方で、情報がいっぱい手に入るが故にあれもいい、これもいいとおしゃれな画像ばかりが溜まりまっていき、画像が膨大になります。さらに集めた画像もデザインやスタイル傾向が違うものがあったりと一つの家に違った要素を合わせるとまとまりの無い家になってしまいます。
　そこでたくさんの画像の中から自分の好きなものと嫌いなものを整理し、かつ好きなものの傾向を４象限マトリックス状に配置し、傾向を視覚化することで自分の好み家のスタイルが明確になり、それによりハウスメーカー担当者との家づくりにスムーズに進められるのではと考えました。
　また同時に家づくりはパートナーの意見も非常に重要なので同じ写真の画像群で好き嫌いを判別すると自分とパートナーとの共通の好みが視覚化され、家づくりあるあるでよく見る夫婦間の意見の相違のよる喧嘩など避けられるという効果も期待されると考えました。
これらの理由からこのアプリを作成したいと思いました。

### 2-2. なぜそれが「気になった」のか
　私は性格上、理屈っぽい性格をしています。人の意見に対しても理由が明確でないと理解に時間がかかったりします。そのような性格であるため、より理屈やデータでデザインの様な曖昧だと感じていた部分の解像度を上げることに魅力を感じた点とパートナーとの衝突も避けられる折衷案を取れるという点に魅力を感じたからです。
家づくり中のパートナーとの意見の相違で喧嘩などはSNSでよく見かけるワードであったことも後押ししています。

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと

１） 画像サンプルばかりが集まって自分たちが共通していいと思うものにたどり着くまでに時間がかかるという点
２） 画像を一つ一つ検索して、好きなものだけ保存してという作業の手間
３） 自分一人で選んだものも傾向がバラバラで一つのまとまりのある方向性に絞りきれないという点

### 3-2. 本当に解決したい課題は何か
１）なぜ？
　　→そもそも自分の好きなジャンルが整理されていない、かつパートナーも好みが整理しきれていない状況で共通点を見つけることが困難であり、本来すり合わせには時間がかかる
　　→画像収集ばかりでデータが膨大になる一方で、それらの整理をする時間をとっていない（取れない）
　　→整理には労力と時間がかかるため
２）なぜ？
　　→画像を検索して気になったものを見つけ、その個別ページに飛んで、その画像を保存するという工程が多いことが手間。かつ保存先もその他の家づくりとは関連のない雑多な画像と同じところに保存されてしまうのでそこから打ち合わせのたびに画像を探すのが手間
　　→作業の工程数が多いことと保存先ルールが明確でないから
　　→作業工程数最適でかつ保存ルールが仕組み化されていないから
３）なぜ？
　　→傾向を分析して、絞るという作業をしていないから
　　→労力と時間がかかるからやらなくなる
　　→労力と時間が短縮する仕組みがないから

→ 本当に解決したい課題：
　家づくりアイデアをまとめる作業において、作業の手間がミニマムで、かつ要因分析という労力のかかる部分の負担を軽減させたい。

## 7. このアプリで実現すること

### MVPで作る機能
- 好み選別＋アイデア画像の保存機能（家づくりアイデア画像が一枚ずつカード形式で表示されるので、それの好き嫌いをすることで画像の保存と好みの分類をしていく）
- AIを使った好みの分析（画像の中の要素（色、形、素材、光、空間）という要素ごとに配分を点数評価させることで好みの画像を分析する）
  - AIでの画像分析はMVP以降の実装に切り替えます。（理由：安定した挙動保証、並びに実装までの時間的配分を考えての判断。ただし、診断自体はGPT-4oによる診断で事前に画像ごとに診断結果のデータを持たせることで MVP開発予定であるため、AI画像診断の品質はMVP後と同程度と推測しています。）
- 傾向マッピング機能（分析した画像たちを４象限マトリックス状にマッピングし、傾向を視覚化する）
- マッピング軸変更機能（４象限の縦軸・横軸はアプリ側で規定した設定のものから任意で変更する機能）
- UMF（User Most Favorite）機能（最もユーザーの好みの傾向が強い選択した画像群だけを抽出する機能）
- UMF機能で得られた画像群に名前をつけて保存する機能（ハウスメーカーとの打ち合わせ時にその画像群を見せることで自分の好みがすぐに伝わる機能）

### 本リリースで作る機能
- パートナーと同じ画像を使って傾向分析する機能（パートナーにも同じ画像群を使って好みの選別をできる様にする機能）
- シンクロ・マップ機能（同じ画像を使って傾向分析したパートナーのマップと自分のマップを重ね合わせることで自分とパートナーとの共通の好みや正反対の好みの傾向を視覚化する機能）
- 材料となる画像の拡充機能（Pinterest APIでは、調査の結果、著作権周りでの運用リスクがあることが発覚しました。MVPではAI生成画像を用いて画像素材を確保し、MVP後に著作権周りも含め問題のない画像素材に切り替えます。）

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- フレームワーク：
  バック/Ruby on Rails 7.2 (DBSCANとの互換性都合)
  フロント/HotWire(Stimulus)
- DB：PostgreSQL
- デプロイ先：Render
- 使用`予定`ライブラリ：
　フロント：
    - Stimulus
    - Anime.js　4.2.x
    - Hull.js 1.0.x
    - Lucide Rails 0.7系
　バック：
    - Devise 5.0.1：アカウント登録
    - ruby-openai (MVP時未実装、MVP時は事前にGPT-4oで判定させた画像をローカルで管理)
    - Active Storage + ImageProcessing
    - DBSCAN または　density-clustering 2.1.0
  CSS:
    - Tailwind CSS　4.x

### 10-2. 技術選定の理由
あくまで上記は予定という前提で選定理由は以下です。
- なぜこの技術を使うのか
  - フロント
    - Stimulus：カリキュラムで使っているかつ、動的なUXがこのアプリにおいてユーザーを惹きつける要素となり得ると考えているため
    - Anime.js：軽量で時間差アニメーションに強いため、今回のアプリでの演出としての要素に強みがあるため
    - Hull.js：マッピングの際の傾向の視覚が良いになるため
    - Lucide Rails：モダンなアイコンセットを活用したいため
  - バック
    - Devise：Railsでのユーザー登録実装を容易にするため
    - ruby-openai：今回GPT -4o/Vision APIとの連携のため
    - Active Storega ＋ ImageProcessing：画像アップロード管理のため
    - DBCAN：マッピングの座標に基づくクラスタリングを容易にするため

- 今回チャレンジしたい点
  - GPT-4o/Vision API実装(API実装初)
  - JSを使ったリッチなユーザー体験の実装
- 不安な点
　- APIの実装などの技術的な初めてな部分
